# ENGINEERING_GUIDE.md — Flexar Hub Web

**Это обязательный к прочтению документ для каждого сабагента.** Прочитай
его целиком до написания кода. Следуй ему буквально. Самодеятельность
запрещена — если чего-то не хватает или гайд противоречит задаче, **не
изобретай, флагни оркестратора**.

Гайд заморожен в Фазе 0. Изменения — только через оркестратора.

---

## 0. Золотые правила (нарушение = задача не принята)

1. **Токены — единственный источник истины.** Ноль «магических»
   значений: цвета, отступы, радиусы, размеры/веса шрифтов, длительности
   анимаций — только через токены (CSS-переменные `var(--…)`).
2. **Стили — только CSS Modules.** Никаких инлайн-`style={{}}`, никакого
   ad-hoc CSS-in-JS, никаких глобальных `.css` кроме разрешённых
   (см. §4). Линт это ловит и красит гейт.
3. **Не пересоздавай существующее.** Перед написанием — проверь
   `COMPONENT_REGISTRY.md`. Если примитив/хук/утилита/тип уже есть —
   используй его. Нужен новый общий примитив — **флагни оркестратора**,
   не делай молча.
4. **Ноль копипасты.** Повторяющаяся логика → общая утилита/хук (через
   оркестратора). Повторяющаяся разметка → компонент.
5. **Ноль ассетов и брендинга из оригиналов.** Ничего из Zulip и из
   старого фронта Flexar/RAGFlow: ни иконок, ни логотипов, ни картинок,
   ни шрифтов, ни цветовых схем. Только `tokens.ts` и собственные ассеты
   Flexar Hub.
6. **Эстетическая согласованность.** Отступы/выравнивания/шрифты/радиусы
   единообразны и симметричны; ничто не «выбивается». Однотипные
   элементы — одинаковый ритм. Сверяйся с соседними компонентами.
7. **Границы.** Трогай только файлы своей задачи. Не редактируй чужие
   компоненты, общие контракты, конфиги. Нужно — флагни оркестратора.
8. **Гейты зелёные.** `npm run typecheck`, `npm run lint`,
   `npm run test`, `npm run build` — зелёные перед сдачей. Красный = не
   сдано.
9. **Не коммить, не мержи, не пушь.** Оставляй изменения оркестратору.

## 1. Стек (зафиксирован, не меняется без оркестратора)

Vite · React 18 · TypeScript strict · React Router · TanStack Query
(server-state) · Zustand (UI-state) · CSS Modules · Vitest +
Testing Library · Playwright · ESLint + Stylelint + Prettier.

Менеджер пакетов — **npm**. Новые зависимости — только через
оркестратора (обоснуй необходимость, не добавляй молча).

## 2. Токены и тема

- Источник значений — `src/theme/tokens.ts`. В нём два слоя:
  **(а)** импортированные Flexar-токены (бренд-цвета light/dark);
  **(б)** `scales` — semantic-шкалы дизайн-системы Flexar Hub
  (заданы оркестратором, PRD-решение). Дополнять токены — **только
  через оркестратора**, с фиксацией здесь и в реестре.
- **Шкалы (`scales` в `tokens.ts`):** `space` (0,1,2,3,4,5,6,8,10,12 —
  ключ ≈ px/4), `radius` (sm/md/lg/xl/full), `fontSize`
  (xs/sm/md/lg/xl/2xl, базовый UI — md=14, xs=11 — микро-лейблы),
  `fontWeight` (regular/medium/semibold/bold), `lineHeight`
  (tight/normal/relaxed), `letterSpacing` (normal/wide),
  `controlHeight` (sm/md/lg), `duration` (fast/base/slow), `shadow`
  (sm/md/lg), `zIndex`, `fontFamily` (base/mono).
- Фаза 0.2 строит **токен-пайплайн**: `tokens.ts` → CSS-переменные в
  `:root` (светлая) и в теме `dark`, плюс типизированный TS-объект темы
  и `ThemeProvider` + хук переключения. **Конвенция имён CSS-переменных
  (пайплайн обязан её соблюсти):** `--space-3`, `--radius-md`,
  `--font-size-md`, `--font-weight-semibold`, `--line-height-normal`,
  `--letter-spacing-wide`, `--control-height-md`, `--duration-base`,
  `--shadow-md`, `--z-modal`, `--font-family-base`; цветовые роли —
  `--color-*` (конкретный список ролей фиксирует пайплайн и заносит в
  `COMPONENT_REGISTRY.md`).
- **До завершения Фазы 0.2 компоненты, зависящие от темы, не пишутся.**
  Точные имена переменных и экспортов сверяй с `COMPONENT_REGISTRY.md`.
- В компонентах: цвета/отступы/радиусы/типографика — **только**
  `var(--token-name)` в CSS Modules. Никогда не хардкодь `#fff`,
  `16px`, `0.5rem` для токенизируемых свойств.
- Разрешённые не-токены в CSS: `transparent`, `currentColor`,
  `inherit`, `none`, `0`, чисто-раскладочные значения (`flex`,
  `grid-template`, `100%`, `auto`) — то, что не является частью
  дизайн-системы.

## 3. Структура файлов и нейминг

Каталоги — по PRD §4.4 (`src/app|theme|api|realtime|stores|domain|
components|features|pages|lib|icons`).

- **Примитивы** (переиспользуемые UI-кирпичи) → `src/components/<Name>/`:
  `Name.tsx`, `Name.module.css`, `Name.test.tsx`, `index.ts` (ре-экспорт).
- **Доменные блоки** (sidebar, message-feed, …) → `src/features/<feature>/`.
- **Экраны** → `src/pages/`.
- **Хуки** → рядом с владельцем или `src/lib/hooks/`; имя `useX`.
- **Утилиты** → `src/lib/`; чистые функции, с unit-тестами.
- **Доменные типы** → `src/domain/`.
- Файлы компонентов — `PascalCase`; хуки/утилиты — `camelCase`; CSS
  Modules — `<Name>.module.css`; классы внутри — `camelCase` (под
  CSS-Modules-маппинг; kebab-проверка стайллинта отключена осознанно).
- Экспорт компонента — **named export** (`export function Button`),
  плюс `index.ts` с ре-экспортом. Без `default export`.

---

## 4. Стилизация

- **Только CSS Modules** (`*.module.css`), импорт `styles from
  './Name.module.css'`, классы `className={styles.foo}`.
- Глобальный CSS — единственный разрешённый: токен-пайплайн в `theme/`
  (объявление CSS-переменных, reset). Больше нигде глобальных стилей нет.
- Инлайн-`style` — запрещён. Динамика — через классы-модификаторы или
  CSS-переменные, проброшенные классом (не атрибутом `style`).
- Stylelint обязывает `var(--…)` для `color`/`background-color`/
  `border-color`, запрещает hex. То же дисциплинируй для отступов,
  радиусов, типографики (даже там, где линт пока не ловит — это правило
  гайда, не только линтера).

---

## 5. Конвенция компонентов

- Пропсы — типизированный `interface NameProps`; явные типы везде, без
  `any`.
- Примитив не лезет в сторы/сеть напрямую — он презентационный, данные и
  колбэки приходят пропсами. Доменные блоки (`features/`) подключают
  сторы/хуки.
- Каждое состояние имеет вид: hover / focus(-visible) / active /
  disabled / selected, и для данных — loading / empty / error /
  один-vs-много / overflow. Реализуй все применимые.
- Иконки — через общий `Icon`-примитив и собственный набор в
  `src/icons/`. Не тащить FontAwesome/иконки Zulip.

---

## 6. Данные: API-клиент, realtime, сторы

- **Сеть — только через `src/api/` клиент.** Не пиши свой `fetch`, не
  ходи на эндпоинты напрямую. Точные методы — в `COMPONENT_REGISTRY.md`
  после Фазы 0.4.
- **Real-time — только через `src/realtime/`.** Не подписывайся на
  события сам.
- **Состояние:** server-state — TanStack Query через предоставленные
  хуки; UI-state — Zustand-сторы из `src/stores/`. Не создавай свои
  сторы для общих сущностей — используй существующие (реестр).
- Доменные типы — из `src/domain/`. Не дублируй типы ответов API в
  компонентах.

---

## 7. Доступность (базовый уровень, обязателен)

- Семантические теги; интерактив — `button`/`a`/`input`, не `div` с
  `onClick`.
- Клавиатура: всё кликабельное — фокусируемо и активируемо с клавиатуры;
  видимый `:focus-visible` (через токены).
- `aria-*` и `label` где нужно; модалки — фокус-трап и `Esc`.
- Контраст — из токенов (они уже выверены).

---

## 8. Тестирование

- Логика (редьюсеры, парсеры, утилиты, сторы) — **unit-тесты Vitest**
  обязательны.
- Компонент — `*.test.tsx` (рендер + ключевые взаимодействия).
- Браузер-протык — делает оркестратор по PRD §7.1 (включая эстетическую
  согласованность). Твоя задача — чтобы протык было чем проходить:
  компонент должен иметь все состояния и быть демонстрируемым (история
  в витрине примитивов / стори-странице, если она заведена).
- Перед сдачей — все гейты зелёные.

---

## 9. Чего НЕ делать (сводка)

- ❌ инлайн-`style`, CSS-in-JS, глобальные стили вне `theme/`
- ❌ хардкод цветов/отступов/радиусов/шрифтов
- ❌ свой `fetch`, свой стор для общей сущности, прямые подписки на события
- ❌ пересоздание того, что есть в `COMPONENT_REGISTRY.md`
- ❌ ассеты/иконки/лого/шрифты из Zulip или legacy-Flexar
- ❌ правка чужих файлов, контрактов, конфигов
- ❌ новые зависимости/примитивы/утилиты молча
- ❌ `default export`, `any`
- ❌ коммит/мерж/пуш

**Не хватает чего-то из этого списка-разрешений? Флагни оркестратора.**
