## 1. Контекст и видение

### 1.1. Что это

**Flexar Hub Web** — веб-фронтенд корпоративного AI-нативного мессенджера
Flexar Hub, написанный **с нуля** на React, использующий **Zulip-сервер
только как бэкенд** (REST API + events API). Сам Zulip-сервер не
модифицируется; его собственный веб-фронт (`web/`) не используется.

### 1.2. Почему с нуля, а не доработка фронта Zulip

Зафиксировано по итогам предыдущей работы:

- Веб-фронт Zulip — **≈133 000 строк** продакшн-кода (`web/src` 144.5k TS/JS,
  `web/styles` 47.8k CSS, `web/templates` 14.5k HBS; ~212k со всеми тестами).
- У Zulip есть **система цвета** (CSS-переменные `--color-*`) — поэтому
  цветовая ретема через оверрайды зашла. Но **системы spacing/layout у него
  нет** — поэтому раскладочная подгонка через CSS-оверрайды выродилась в
  whack-a-mole: каждый патч вскрывал следующее рассогласование.
- Вывод: чужой layout нельзя привести к своей дизайн-системе оверрайдами.
  Нужен фронт, где spacing/layout/типографика **выводятся из токенов**, а не
  патчатся.

### 1.3. Почему «чистый лист» — оправданный риск

- Нет легаси: ни мешанины RAGFlow+Flexar (текущий фронт Flexar тоже под
  рерайт), ни 20 лет истории Zulip.
- Zulip-сервер остаётся стабильным, проверенным бэкендом — мы переписываем
  **только UI-слой**, не трогая хранение, доставку, права доступа.
- Чистый рерайт на современном стеке реалистично в разы компактнее
  оригинала (ориентир — **15–30k строк** на крепкий messenger), потому что
  не тащит легаси; но объём *поведения* к воспроизведению от этого не
  уменьшается — это недели-месяцы итеративной работы.

### 1.4. Дизайн-система

Единый источник истины — **`tokens.ts`** (взят из репозитория Flexar:
`web/src/theme/tokens.ts`; на старте — копия `flexar-web/tokens.seed.ts`).
Токены концептуально **общие** для Flexar и Flexar Hub; разрешено
добавлять новые токены по мере необходимости (с фиксацией в
`ENGINEERING_GUIDE.md`). Компоненты пишутся **с нуля** (без Ant Design,
без кода Zulip).

### 1.5. Не-цели (out of scope)

- Не модифицируем Zulip-сервер и его `web/`.
- Не делаем 1:1 паритет фич в первой версии — паритет это дорожная карта.
- Не реализуем серверные интеграции/вебхуки/биллинг.
- Мобильное приложение — отдельный трек (Flutter, см. roadmap форка).
- AI-фичи (auto-topicing, агенты) — отдельный трек поверх готового каркаса.

## 2. Принципы разработки (обязательны к соблюдению)

1. **Токены — единственный источник истины.** Ни одного «магического»
   px/цвета/радиуса в компонентах: всё через токены/CSS-переменные.
2. **Компоненты с нуля.** Никакого Ant Design, никакого копирования кода
   Zulip. Только токены берём.
3. **Каждый компонент протыкивается через браузер** до перехода к
   следующему. «Оттыкан» имеет формальное определение (см. §7).
4. **Инкрементально.** Сделал каркас → протыкал → зафиксировал. Добавил
   компонент → протыкал → зафиксировал. Никаких «соберём всё, потом
   проверим».
5. **Оркестрация, не кодинг в одиночку.** Реализация — параллельными
   сабагентами по спецификациям; главный поток оркеструет, интегрирует,
   протыкивает, держит общую картину (см. §6).
6. **Контракты заморожены до параллельной работы.** Общие типы, API-клиент,
   токен-система, конвенции API компонентов — определяются и стабилизируются
   ПЕРВЫМИ. Иначе параллельные агенты конфликтуют.
7. **Строгие гейты.** TypeScript strict, ESLint, Prettier, типчек, билд,
   unit-тесты — зелёные на каждом шаге. Красный гейт = стоп.
8. **Каждое состояние имеет вид.** loading / empty / error / hover / focus /
   active / disabled / overflow — для каждого компонента, в светлой и тёмной
   теме.
9. **Ноль велосипедов, ноль копипасты, ноль инлайн-стилей.** Это «чистый
   современный рерайт», а не сборка из кусков. Механизм принуждения — §6.5:
   единый инженерно-дизайн-гайд + живой реестр уже реализованного +
   линт-правила, которые *механически запрещают* инлайн-стили, хардкод
   значений и обход токенов. Сабагент не импровизирует — он следует гайду.

## 3. Технический стек

| Слой | Выбор | Обоснование |
|---|---|---|
| Сборка | **Vite** | Быстрый dev-сервер, простой прод-билд |
| UI | **React 18 + TypeScript (strict)** | Совпадает с экосистемой Flexar |
| Роутинг | **React Router** | URL ↔ narrow-состояние |
| Server-state | **TanStack Query** | Кэш, ретраи, инвалидация для REST |
| UI-state | **Zustand** | Лёгкие сторы (narrow, compose, UI) |
| Стили | **CSS Modules + CSS-переменные из токенов** | Без рантайм-CSS-in-JS; токены → `:root` vars |
| Иконки | Собственный icon-компонент (SVG-спрайт) | Свой набор, не FontAwesome Zulip |
| Real-time | **Zulip events API** (register queue + long-poll) | Штатный механизм Zulip |
| API-клиент | Типизированная обёртка над `fetch` | Без тяжёлых SDK |
| Unit-тесты | **Vitest + Testing Library** | Логика, редьюсеры, утилиты |
| E2E/браузер | **Playwright** + ручной «протык» через браузер | Критические флоу + визуальная проверка |
| Линт/формат | **ESLint + Prettier** | Гейты |
| CI | GitHub Actions | Типчек + линт + unit + билд на каждый PR |

**Размещение (решено):** в текущем репозитории `flexar-zulip`, каталог
`flexar-web/`, отдельная ветка `flexar-web` (создана от `origin/main` —
не несёт коммитов CSS-ретемы из `flexar/main`). Путь:
`/Users/alexander/projects/flexar-zulip/flexar-web`.

## 4. Архитектура

### 4.1. Слои (снизу вверх)

```
  tokens.ts  ──►  CSS-переменные + TS-тема (ThemeProvider)
       │
  api/        ──  типизированный клиент Zulip REST + events
       │
  realtime/   ──  register queue, long-poll /events, reconnect/backoff,
       │          диспетчер событий → редьюсеры
  stores/     ──  server-state (TanStack Query) + UI-state (Zustand):
       │          realm, users, streams/subscriptions, messages, narrow,
       │          compose, unread, presence
  components/ ──  примитивы (Button, Input, Avatar, Modal, …) на токенах
       │
  features/   ──  доменные блоки (sidebar, message-feed, compose, search,
       │          reactions, settings, …)
  pages/      ──  Login, App-shell (3 колонки), Settings overlay
       │
  app/        ──  роутинг, провайдеры, точка входа
```

### 4.2. Цикл реального времени

1. После логина — `POST /api/v1/register` → начальное состояние + `queue_id`.
2. Long-poll `GET /api/v1/events?queue_id=…&last_event_id=…`.
3. Каждое событие → диспетчер → редьюсер соответствующего стора.
4. `BAD_EVENT_QUEUE_ID` / разрыв → переустановить очередь с backoff.
5. Оптимистичные действия (отправка, реакция) — локальное эхо →
   реконсиляция по приходящему событию.

### 4.3. Аутентификация

- `POST /api/v1/fetch_api_key` (email + password) → `api_key`.
- Ключ + email хранятся (httpOnly через прокси предпочтительно; на этапе
  каркаса — `localStorage`, с явным TODO на хардненинг).
- Все запросы — Basic auth (`email:api_key`).
- Logout — очистка, дерегистрация очереди.

### 4.4. Структура каталогов (заморозить в Фазе 0)

```
flexar-web/
├── PRD.md
├── ENGINEERING_GUIDE.md, COMPONENT_REGISTRY.md
├── package.json, tsconfig.json, vite.config.ts, .eslintrc, …
├── src/
│   ├── app/            точка входа, роутинг, провайдеры
│   ├── theme/          tokens.ts, генерация CSS-переменных, ThemeProvider
│   ├── api/            REST-клиент, типы ответов Zulip
│   ├── realtime/       events queue, диспетчер, редьюсеры
│   ├── stores/         Zustand-сторы + хуки TanStack Query
│   ├── domain/         доменные типы (Message, Stream, User, Narrow, …)
│   ├── components/     примитивы UI
│   ├── features/       доменные блоки
│   ├── pages/          экраны
│   ├── lib/            утилиты (formatting, narrow-парсер, markdown-hydrate)
│   └── icons/          SVG-иконки
├── tests/              Vitest unit
└── e2e/                Playwright
```

## 5. Контракт интеграции — поверхность Zulip API

Минимальный набор эндпоинтов, который надо обернуть типизированно
(заморозить интерфейсы в Фазе 0–1):

| Область | Эндпоинты |
|---|---|
| Auth | `POST /fetch_api_key`, `GET /server_settings` |
| Начальное состояние | `POST /register` (realm, users, streams, subscriptions, unread, presence) |
| События | `GET /events`, `DELETE /events` (дерегистрация) |
| Сообщения | `GET /messages`, `POST /messages`, `PATCH /messages/{id}`, `DELETE /messages/{id}`, `GET /messages/{id}` |
| Флаги | `POST /messages/flags` (read, starred, …) |
| Реакции | `POST /messages/{id}/reactions`, `DELETE /messages/{id}/reactions` |
| Каналы | `GET /streams`, `GET /users/me/subscriptions`, `POST /users/me/subscriptions`, `DELETE …`, `PATCH /streams/{id}` |
| Топики | `GET /users/me/{stream_id}/topics`, `POST /messages` (move/resolve) |
| Пользователи | `GET /users`, `GET /users/me`, `GET /users/{id}`, `PATCH /settings` |
| Typing | `POST /typing` |
| Presence | `POST /users/me/presence`, `GET /realm/presence` |
| Загрузки | `POST /user_uploads` |
| Черновики | `GET /drafts`, `POST /drafts`, `PATCH /drafts/{id}`, `DELETE /drafts/{id}` |
| Отложенные | `GET /scheduled_messages`, `POST /scheduled_messages`, … |
| Эмодзи | `GET /realm/emoji` |
| Инвайты | `POST /invites` (для управления) |

**Замечание про markdown:** Zulip отдаёт сообщения уже как
`rendered_content` (готовый HTML). Фронт **не парсит markdown** — он
**санитизирует и гидрирует** серверный HTML (подсветка кода, KaTeX,
эмодзи, спойлеры, превью). Это упрощает Фазу 1.

## 6. Модель оркестрации сабагентами

### 6.1. Роли

- **Главный поток (оркестратор)** — я. Не пишу код. Декомпозирует на
  задачи-спеки, запускает сабагентов, интегрирует результат, прогоняет
  гейты, протыкивает в браузере, ведёт чеклист и общую картину.
- **Сабагенты-исполнители** — пишут код по выданной спецификации в
  изолированном git-worktree.

### 6.2. Спецификация задачи для сабагента (шаблон)

Каждая задача выдаётся как самодостаточная спека. **Первой строкой
каждой спеки — обязательное предписание:** «прочитай `ENGINEERING_GUIDE.md`
и `COMPONENT_REGISTRY.md` целиком до написания кода; следуй им буквально;
самодеятельность запрещена». Далее:
- **Цель** — что за компонент/фича.
- **Контракт** — props/API, типы (ссылка на замороженные общие типы).
- **Что уже есть (использовать, не пересоздавать)** — выписка из
  `COMPONENT_REGISTRY.md`: какие примитивы/хуки/утилиты/токены агент
  ОБЯЗАН переиспользовать.
- **Зависимости** — какие токены/примитивы/сторы/эндпоинты использует.
- **Acceptance criteria** — поведение, состояния.
- **Браузер-чеклист «протык»** — что проверить визуально/интерактивно.
- **Не делать** — границы: не трогать чужие файлы; не создавать новые
  примитивы/утилиты (нужен новый — флагнуть оркестратора, не делать
  молча); не писать инлайн-стили; не хардкодить значения.

### 6.3. Правила параллелизации

1. **Фаза 0 (фундамент) — почти последовательно.** Токен-система,
   API-клиент, общие типы, конвенции компонентов, app-shell — это
   замороженные контракты. Их делаем первыми и аккуратно; на этом этапе
   параллелизм минимальный (только независимые примитивы после того, как
   конвенции зафиксированы).
2. **После заморозки контрактов — широкий параллелизм.** Независимые
   компоненты/фичи — параллельные сабагенты (напр. `Button`, `Input`,
   `Avatar`, `Modal` — 4 агента разом).
3. **Изоляция.** Каждый параллельный сабагент — в своём git-worktree, чтобы
   не было коллизий по файлам.
4. **Зависимые фичи — последовательно** или с явной синхронизацией
   (compose зависит от typeahead; message-actions зависит от message-row).
5. **Интеграция — только через главный поток.** Сабагент не мержит сам;
   оркестратор интегрирует, гоняет гейты, протыкивает.

### 6.4. Цикл итерации

```
спека → сабагент(ы) пишут → оркестратор интегрирует →
  типчек+линт+unit+билд → браузер-протык → ✔ чеклист → следующий шаг
```

Красный гейт или непройденный протык = шаг не закрыт, идём чинить, дальше
не двигаемся.

### 6.5. Механизм согласованности (анти-велосипед, анти-копипаст, анти-инлайн)

Параллельные сабагенты не делят память — поэтому согласованность нельзя
оставлять на добрую волю. Три слоя принуждения:

**Слой 1 — `ENGINEERING_GUIDE.md` (единый гайд, читают ВСЕ).**
Создаётся в Фазе 0 как первоклассный артефакт, до любой параллельной
работы. Каждая спека сабагента обязывает прочитать его целиком. Содержит:
- как потреблять токены (только CSS-переменные / TS-тема; ни одного
  «магического» значения);
- **стилизация — только CSS Modules**; `style={{…}}` инлайн — запрещён
  полностью; никаких ad-hoc CSS-in-JS;
- где живут примитивы, как они называются, конвенция их props;
- структура файлов, нейминг (компоненты, хуки, утилиты, типы, тесты);
- как потреблять API-клиент и сторы (не свой `fetch`, не свой стор);
- паттерны состояний (loading/empty/error), паттерны доступности;
- что делать, если «нужного нет» — **флагнуть оркестратора, не
  изобретать**.

**Слой 2 — `COMPONENT_REGISTRY.md` (живой реестр реализованного).**
Ведёт оркестратор; обновляется после каждой интеграции. Реестр всего, что
уже существует: примитивы, хуки, утилиты, сторы, токены — с сигнатурами и
одной строкой назначения. Каждая спека сабагента включает релевантную
выписку из реестра с предписанием «использовать это, не пересоздавать».
Так агент физически знает, что Button/Input/useNarrow/… уже есть.

**Слой 3 — линт-правила (механический запрет, сильнейший слой).**
Настраиваются в Фазе 0, в гейтах CI:
- ESLint: запрет инлайн-`style` на DOM- и компонентных элементах
  (`react/forbid-dom-props`, `react/forbid-component-props` → `style`);
  запрет «голых» цветовых/числовых литералов в TSX.
- Stylelint: запрет hex/`rgb()`-литералов (как `color-no-hex` у Zulip);
  запрет «магических» px вне токен-шкалы; только `var(--token)` для
  цветов/отступов/радиусов/типографики.
- Запрет импорта «сырых» значений мимо токен-модуля.
- (Опц.) детектор дублей: проверка, что новый файл не повторяет имя/экспорт
  уже существующего примитива/утилиты.

**Гейт интеграции (оркестратор, на каждом мердже сабагента):**
проверяет — не создан ли дубль того, что есть в реестре; нет ли инлайн-
стилей и хардкода (линт это ловит, но оркестратор смотрит и глазами); все
значения из токенов; имена/структура по гайду; **эстетическая
согласованность по §7.1** (отступы/выравнивания/шрифты единообразны и
симметричны). Не прошло — назад на доработку, в чеклист не идёт.

## 7. Протокол тестирования

### 7.1. Определение «оттыкан» (browser-tested)

Компонент считается оттыканным, когда в браузере проверено:
- рендерится без ошибок консоли;
- все интерактивные состояния: hover, focus (с клавиатуры), active,
  disabled, selected;
- все состояния данных: loading, empty, error, единичный/много, overflow
  (длинные строки/имена);
- **светлая и тёмная тема** — обе;
- клавиатурная навигация и фокус-видимость;
- значения соответствуют токенам (выборочный замер `getComputedStyle` /
  `getBoundingClientRect`, не «на глаз»);
- **эстетическая согласованность** — отступы, выравнивания, размеры и веса
  шрифтов, радиусы единообразны и симметричны; ничто не «выбивается». Нет
  разнобоя в gap/padding между однотипными элементами, нет асимметрий, нет
  шрифтов «не того» размера/веса. Сверять с соседними и аналогичными
  элементами; при сомнении — замер, не «на глаз». (Именно на этом
  спотыкалась CSS-ретема Zulip — здесь это контролируется системно, на
  каждом протыке.)
- адаптив: ключевые брейкпоинты (десктоп 1920 / ноут 1280 / планшет /
  телефон 480) — там, где компонент влияет на раскладку.

### 7.2. Уровни тестов

- **Unit (Vitest)** — редьюсеры событий, narrow-парсер, форматтеры,
  утилиты, логика сторов. Обязательно для всего, что не «чистая вёрстка».
- **Браузер-протык** — каждый компонент, по чеклисту §7.1, через
  Claude-in-Chrome / preview. Ручной, но фиксируется в общем чеклисте.
- **E2E (Playwright)** — критические сквозные флоу: логин, отправка
  сообщения, переключение narrow, реакция, поиск. Прогоняются на каждой
  фазе.

### 7.3. Инкрементальный гейт

Никакой компонент/фича не считается готовым и не идёт в общий чеклист,
пока не прошёл: гейты (типчек/линт/unit/билд) + браузер-протык. Каркас
протыкивается до добавления первого компонента; каждый компонент — до
следующего.

### 7.4. Окружение тестирования (решено)

- **Фронт — локально:** Vite dev-сервер на `localhost:5173`. Протык — через
  браузер (Claude-in-Chrome) по §7.1; Playwright-флоу — против него же.
- **Бэкенд — задеплоенный стенд:** `https://95.84.162.15:8843` (Zulip).
  Локальный Zulip не поднимаем — стенд и есть настоящий бэкенд.
- **Связка — Vite dev-proxy** (`vite.config.ts → server.proxy`): `/api/*`
  проксируется на стенд. Это убирает CORS (для браузера всё same-origin) и
  снимает самоподписанный TLS (`secure: false`); `changeOrigin: true` —
  корректный `Host` для резолва реалма. Логин, REST и events long-poll —
  всё через прокси.
- **Данные:** тестируем на боевых данных стенда. Неразрушающий протык
  (отправка в тестовые каналы, narrow, реакции) — норма. Разрушающие
  действия (удаление, деактивация) — только на отдельных тестовых
  сущностях. Нужен **отдельный тестовый аккаунт** на стенде (§12.4).
